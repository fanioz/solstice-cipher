## Context

### Problem Statement
We need a performant, instant-feedback system to propagate light beams in 3D space, evaluating reflections, color splitting, and cryptographic logic gates as the player rotates interactive nodes.

### Constraints
- Must guarantee determinism.
- Must avoid one-frame physics syncing delays (instant feedback).
- Must interact correctly with both interactive `OpticNode` objects and procedural static walls.
- Must not crash if players create an infinite optical loop (e.g., a 4-mirror square).

## Decision

We will use native **PhysicsDirectSpaceState3D Raycasting (Jolt Physics)** as the core method for runtime beam propagation. 

When a source node emits light, or an `OpticNode` is rotated, the `BeamPropagationManager` will perform an iterative loop of physics raycasts. Abstract graph traversal (relying solely on known puzzle paths) was rejected because it cannot determine collision endpoints if a player intentionally or accidentally rotates a mirror off-path into a wall. By using native raycasting, beams will dynamically evaluate intersections against the procedural geometry generated by ADR-0005.

**Loop Mitigation:** We will implement a strict integer `max_bounces` limit on the iterative raycast loop. This guarantees that infinite optical loops or recursive reflections are safely terminated without triggering GDScript stack overflows.

**Collision Layers & Masks:** The raycast queries will strictly filter against specific Collision Masks (e.g., `LAYER_OPTICS`, `LAYER_WALLS`). Dynamic physics props (e.g., physics debris or player character bodies) will be ignored by the logic raycast to ensure puzzle determinism, though visual intersection handling may differ.

**Float Stability:** To mitigate Jolt float drift across deep raycast chains, intersection normals and positions will be explicitly evaluated using `Vector3.is_equal_approx()` and snapped to a fixed precision (`Vector3.snapped()`) before computing reflection vectors.

All interactive nodes will inherit from an `@abstract class_name OpticNode`. Visual rendering of the beams will be decoupled from logic: `MeshInstance3D` cylinders and `GPUParticles3D` will be stretched between the mathematically confirmed collision points.

### Key Interfaces
- `@abstract class_name OpticNode extends Node3D`
- `class_name BeamPropagationManager extends Node`
- `func update_beams() -> void`
- `func _trace_beam(origin: Vector3, direction: Vector3, initial_color: Color) -> void`

## Alternatives Considered

### Alternative 1: Abstract Graph Traversal (Pure Data)
- **Description**: Trace the mathematical node graph purely in data, bypassing the physics engine for logic.
- **Pros**: Extremely fast; perfectly deterministic.
- **Cons**: Completely fails to determine endpoints when a node is rotated away from known optical paths towards a procedural wall. Requires reinventing bounding box intersection math in pure GDScript.
- **Rejection Reason**: Unmanageable technical complexity duplicating the engine's built-in collision capabilities.

## Consequences

### Positive
- Jolt physics handles high-frequency raycasting with extreme performance.
- Seamlessly resolves collisions against dynamic interactions and procedurally generated walls.
- Strict `max_bounces` eliminates stack overflow risks completely.

### Negative
- Raycast calculations are slightly heavier than pure data traversal.
- Careful tuning of collision layers is required to ensure beams do not accidentally hit unintended physics objects.

### Risks
- **Jolt Floating Point Edge Cases**: `PhysicsDirectSpaceState3D.intersect_ray` returning slightly inaccurate normals at shallow 15-degree angles.
  - *Mitigation*: Aggressively snap the returned intersection normals and positions to the known 15-degree mathematical grid increments before recalculating the next bounce.
